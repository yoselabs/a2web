## Context

The 2026-07-09 Koçtaş re-probe (see `BACKLOG.md`) showed a2web computing a strong, correct answer (JSON-LD `Product.offers.price`) on the raw tier, then forcing an unnecessary browser escalation anyway (gate). It also surfaced a second, related gap in `fetch_raw`'s display pick, which this change attempted and then reverted — see the postscript at the end of this document. Both fixes were meant to reuse a signal a2web already computes — `ContentCandidate.answer_bearing` — at a point that doesn't currently consult it fully; only the gate half survived implementation.

This is not green-field: `quality-gate` already has a related, deliberately-scoped requirement ("Answer-bearing structured content exempts a bare `length_floor` from failure") whose scenarios explicitly assert that anti-bot verdicts are **NOT** masked by an embedded payload, even a strong one — "a genuine SPA shell therefore continues to escalate to the browser tier even when it embeds a stub structured payload." That decision was scoped to the **bare `length_floor`** case: content is thin, and the concern was a stub/decoy payload gaming a thin, ambiguous response. Our case is different in kind, not degree: content is well above `LENGTH_FLOOR` (1343 chars of real prose, not a shell) and the candidate is not a stub — `is_answer_bearing` already requires a preferred schema.org `@type` with ≥3 populated fields. The existing requirement's code path (the bare-`length_floor` promotion) never even reaches the Koçtaş case, because `akamai_bmp`/`turnstile` are checked earlier in `evaluate()` and are length-independent — they fire before any length_floor branch is reached, on both thin and full pages alike. So this change adds a **second, narrower, textually-adjacent exemption**, not a reversal of the first. Design and specs must make that boundary explicit so a future reader doesn't conflate the two.

## Goals / Non-Goals

**Goals:**
- Stop paying the browser-escalation tax (latency + confidence downgrade) on `akamai_bmp`/`turnstile`-marked pages that already yielded a strong, answer-bearing structured candidate on the raw tier.
- Preserve every existing quality-gate guarantee for pages that do NOT have a strong answer-bearing candidate — in particular, do not weaken `anubis` / `alibaba_baxia` / `cf_iuam` / generic `block_page_detected` handling, and do not touch the existing bare-`length_floor` promotion's scenarios.

**Non-Goals:**
- Not touching cookie plumbing, `partial` status, per-domain wall memory, or any envelope/wire shape (the superseded 2026-07-07 backlog items).
- Not changing `is_answer_bearing`'s definition or strictness (owned by the `json_in_html` shelf package).
- Not adding a new operator hint or diagnostics field — existing `Diagnostic`/`subsystem` plumbing is enough to audit this from the trace (open question below).
- Not touching `anubis`, `alibaba_punish`, `cf_iuam`, `search_captcha`, or the generic `block_page_detected` branches at all — only the two length-independent branches (`akamai_bmp`, `turnstile`).
- **Not fixing `fetch_raw`'s display-pick gap in this change** — attempted, reverted; see postscript. Deferred to `BACKLOG.md` for a future change with its own design pass.

## Decisions

**D1 — The exemption lives in `evaluate()` itself, as a new bool parameter, mirroring the existing `structured_answer` parameter.** `packages/block_detector.py` can't import `a2web.<domain>` (tach.toml boundary) and doesn't see `ContentCandidate` objects. **Implementation-time correction**: the "existing `structured_answer` parameter" this design referred to actually lives on the domain-level `evaluate()` *wrapper* in `src/a2web/fetcher.py` (not the pure package function in `block_detector.py`, which has no such parameter). That wrapper already accepts `structured_answer: bool` and already does the analogous bare-`length_floor` promotion after calling the package's `_package_evaluate`. This is the correct existing seam — it's exactly the domain-aware post-processing layer tach.toml's boundary implies must exist somewhere, and it already existed for this purpose. The new branch was added there, reusing the same computed bool (same call site, same value) rather than inventing a second signal or touching `block_detector.py` at all.
  - *Alternative considered*: check `structured_answer` in `_phase_gate_and_escalate` after `evaluate()` returns, overriding its verdict there. Rejected — this would split "what does this response mean" logic across two files/layers for no benefit, and `evaluate()`'s docstring and tests are already the canonical home for every other marker branch.

**D2 — Scope the new branch strictly to `akamai_bmp` and `turnstile`, gated on `len(content_md) >= LENGTH_FLOOR` in addition to `structured_answer`.** The length check keeps this exemption from ever touching the deliberately-preserved thin/stub-anti-bot scenarios (`test_anubis_marker_with_short_content`-style cases) — those stay exactly as documented. Only `akamai_bmp`/`turnstile` are length-independent today; `anubis` already has its own length check baked in (`len(content_md) < LENGTH_FLOOR` in its `if`), so it's structurally unaffected regardless. `alibaba_punish`/`cf_iuam`/`search_captcha`/generic block-page patterns are untouched — different threat shape (page-replacement interstitials, not "defended but functional" sites).
  - *Alternative considered*: exempt regardless of length (i.e., trust `structured_answer` unconditionally for `akamai_bmp`/`turnstile`). Rejected — a thin akamai_bmp page with a lucky stub payload is exactly the adversarial shape the existing bare-`length_floor` requirement was written to guard against; keeping the length floor as a co-requirement costs nothing and preserves that guard.

**D3 — When the exemption fires, verdict becomes `Verdict.ok` with `subsystem=None` (identical shape to the existing bare-`length_floor` promotion), and no escalation is emitted.** Callers (`_phase_gate_and_escalate`) already branch on `resolved_verdict()`/`gate_result.escalation`, so returning a plain `ok` with no escalation is sufficient — no new `Verdict` enum member, no new `EscalationSignal` reason needed.

**D4 — REVERTED. See postscript.** The original plan was to drop the `len(prose_md) < LENGTH_FLOOR` gate on `_pick_display_candidate`'s `answer_bearing` short-circuit, reasoning that only `json_synth` ever sets `answer_bearing=True` so it couldn't regress the "prose beats longer `record_synth`" or "article never clobbered" scenarios. That reasoning was correct about `record_synth` but missed a different failure mode: `answer_bearing` is also `True` for plenty of `json_synth` candidates that coexist with substantial, correct prose — see postscript.

## Risks / Trade-offs

- **[Risk] A site could serve a manipulated/stale JSON-LD payload specifically to bot traffic while still passing `is_answer_bearing`'s strictness, and this change would now accept it without ever rendering via a real browser.** → Mitigation: `is_answer_bearing` requires a preferred schema.org `@type` with ≥3 populated fields sourced from `ld_json`/`microdata` — the same markup search engines (Google Shopping, Bing) consume for rich snippets. A site corrupting this for "bot" traffic risks corrupting its own SEO surface for the crawlers it actually wants, making this a weak attack surface in practice. No change to `is_answer_bearing` itself is in scope here if evidence later shows otherwise.
- **[Risk] Silent behavior change is hard to audit later ("why didn't this escalate?").** → Mitigation: the existing `Diagnostic` trace already records `subsystem`/`verdict` per stage; confirm (task-level) that a distinguishable trace remains — e.g., the diagnostic can still note `akamai_bmp` was matched but exempted, rather than reading identical to a page with no marker at all. Decide exact shape during implementation (see Open Questions).
- **[Risk] Regressing the two existing marker tests (`test_akamai_bmp_marker_suggests_browser`, `test_turnstile_marker_suggests_browser`).** → Mitigation: both call `evaluate()` without a `structured_answer` argument today; default it to `False` so existing call sites and tests are unaffected, and add new tests for the `structured_answer=True` + above-floor case alongside them.
- **[Trade-off] Two similarly-shaped exemptions (bare-`length_floor` promotion vs. this new above-floor anti-bot exemption) now live in `evaluate()`, which slightly raises the function's branch count.** → Accepted: they answer genuinely different questions (thin-page rescue vs. skip-pointless-escalation) and keeping both in `evaluate()` is more legible than splitting gate logic across files (see D1).

## Migration Plan

No data migration. Pure pipeline-logic change behind existing tests — ship via normal `make check` gate. No feature flag (project convention: don't use feature flags when the code can just change). Rollback is a plain revert if a live-network `make bench` run (recommended after this change per the bench-worthiness criteria: extraction pipeline + tier routing) shows a quality regression.

## Postscript (implementation-time revision)

D4 (the `_pick_display_candidate` fix) was implemented, then reverted after `make check` failed 23 tests, most rooted in one cause: `tests/capabilities/tier_pipeline/test_fetcher.py::test_blog_fixture_yields_real_envelope`. The `blog.html` fixture embeds a routine `Article` JSON-LD block (`headline`, nested `author`, `datePublished`, `articleSection` — 4 populated fields). `Article`/`NewsArticle` are in `json_in_html._PREFERRED_LD_TYPES` alongside `Product`/`LocalBusiness`/etc., so this candidate is `answer_bearing=True` by the same rule that correctly flags Koçtaş's `Product` payload. An unconditional "answer_bearing beats prose" rule can't tell "a Product schema that IS the answer, on a page whose prose is irrelevant boilerplate" (Koçtaş) apart from "an Article schema that is routine SEO metadata, on a page whose prose IS the answer" (any normal blog/news article) — both are `answer_bearing=True`, but the correct display pick is opposite in each case. A second, pre-existing test (`test_above_floor_prose_keeps_display_over_structured`, `tests/capabilities/quality_gate/test_structured_answer_exemption.py`) independently encodes the same guarantee for a `Product` schema embedded in a genuine article — this was deliberate prior behavior, not an oversight this change could safely override.

`answer_bearing` is calibrated for the gate's use case (rescue a thin/near-empty page using structured data as its only real source, where there is no substantial prose to lose) — it was never validated as a general-purpose "structured beats prose" display signal, and this postscript is the evidence that it doesn't generalize that way. A future fix needs either (a) a `@type`-level split (treat `Product`/`LocalBusiness`/`ContactPoint`/`Event` — "entity" schemas that rarely co-occur with substantial unrelated prose — differently from `Article`/`NewsArticle`/`ItemList`/`BreadcrumbList` — "editorial" schemas that routinely do), which requires plumbing the schema `@type` onto `ContentCandidate` (currently only a bool), or (b) an actual prose-quality signal (e.g., threading trafilatura's own extraction confidence `score`, already computed in `_ExtractResult.score` but currently dropped when building the `ContentCandidate`, onto the candidate and gating on it). Both are bigger than this change's scope; deferred to `BACKLOG.md`.

`_pick_display_candidate` and `tests/capabilities/extraction/test_extraction_ladder.py` were reverted to their pre-change state; only the gate exemption (Decisions D1-D3) shipped.
