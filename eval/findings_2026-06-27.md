## Decision: keep two as a fast→robust ladder (not one)

The engines are **complementary, not strictly ranked**:

- The Chromium drop-ins (patchright, rebrowser) are **fast (~1.4s)** but **fail the
  Trendyol + Hepsiburada SPAs** — the exact incident class this work targets
  (blocked / empty extraction).
- **zendriver (CDP) reads them** (Trendyol md=3399, Hepsiburada md=7084) and is the
  only backend ≥4/5 on SPA-read, but is **~4-5x slower** (per-render-launch v1).
- **nowsecure-cf** (full Cloudflare challenge) defeats *all three* — out of scope
  (needs a residential proxy / solver).
- **rebrowser is the strict loser**: ties patchright on reads, worse on robustness
  (25s timeout on Cloudflare).

**Outcome:** keep **patchright** (fast first rung, `browser` tier) + **zendriver**
(robust second rung, `browser_robust` tier); **prune rebrowser**. Wired as a
two-tier fast→robust escalation on the *existing* gate→playbook loop (the
`gate_browser_signal` rule, cap widened 1→2). Camoufox gated off (#625
unreleased); `playwright`/`camoufox` deps dropped.

## Phase B (LLM answer-quality)

Deferred. The render-layer result was decisive on the incident class — zendriver
reads the SPAs patchright cannot, so an LLM-quality pass would not change the
keep-both decision. Re-open only if answer quality on the rendered DOM is suspect.

## Follow-up

- **zendriver per-render-launch is the speed cost.** A shared-browser pool (per-host
  reuse, like the Playwright backend) would close most of the ~4-5x gap. Tracked in
  BACKLOG. Browser is the escalation tier, so the cost only bites when the fast rung
  can't read the page.
