# Output benchmark — findings, 2026-05-22

First run of the re-runnable output benchmark (`make bench`). Corpus:
`eval/corpus.yaml` — 10 URLs across listing / comments / clean / gated / spa.
Provider: `claude-code` (OAuth subscription). Judge: `claude-sonnet-4-6`.
Run: 30 cells, $2.06, 5m20s wall. Run dir (regenerable, gitignored):
`eval/runs/2026-05-22_124002/`.

## Headline — four axes, per system

| System | quality | env tokens | clarity | contract | next_links |
|---|---|---|---|---|---|
| webfetch_baseline | 2.90 | 167 | 3.00 | n/a | n/a |
| a2web_detail | 4.10 | 2554 | 1.67 | 10/10 | 4.00 |
| a2web_extract | **4.40** | 430 | **3.67** | 10/10 | 3.67 |

`a2web_extract` (the `ask` tool) is the strongest system: highest answer
quality, highest clarity, full data-contract conformance, and a lean
envelope — all four axes at once.

## Where a2web wins

- **Answer quality:** `a2web_extract` +1.5 over WebFetch (4.40 vs 2.90),
  `a2web_detail` +1.2. a2web won 8/10 URLs (extract) and 7/10 (detail);
  WebFetch 5/10.
- **The gated + SPA edge is real.** WebFetch scores **0.0** on both the
  gated class (medium-tag — HTTP 403) and the SPA class (react.dev —
  HTTP 404 to plain httpx); a2web scores **5.0** on both. WebFetch reach
  6/10 vs a2web 9/10. This is exactly the failure mode a2web exists for.
- **Clarity:** `a2web_extract` 3.67 vs WebFetch 3.00. The server-side
  extracted answer is more directly actionable than WebFetch's raw text.
- **Token cost of `ask` is near-WebFetch.** `a2web_extract`'s lean
  `AskResponse` envelope averages 430 tokens — only +263 over WebFetch's
  167 — while delivering +1.5 quality. The envelope diet works.
- **Data contract:** 10/10 conformant for both a2web systems, under
  `debug=False` and `debug=True`. The v0.14 deviation/debug rules hold on
  every URL.
- **next_links:** scored 4.00 / 3.67 on listing URLs that produced
  candidates. WebFetch has no equivalent.

## Where a2web loses / needs work

- **`a2web_detail` clarity collapses to 1.67** (vs WebFetch 3.00) and its
  envelope balloons to 2554 tokens. Handing the agent the full page
  markdown is expensive and hard to act on — this is the counterfactual
  that justifies `ask` over `fetch_raw`. Detail mode should stay a
  fallback, not a default.
- **Comment threads are weak for everyone.** The `comments` class scores
  ≤2.5 across all systems. `reddit-comments` **fetch-errored on both
  a2web systems**: `tier=raw verdict=length_floor`.
  - *Diagnosis (corrected after probe):* the corpus URL
    (`/r/programming/comments/1ka1bxv/…`) is a **deleted Reddit post**, not
    a live thread. The Reddit handler probed `.json` (404) + `old.reddit`
    (404) and correctly returned `not_found`; the raw tier then got a
    200 + ~8 KB SPA shell → gate `length_floor`. A live Reddit thread
    fetches cleanly via the same handler (12 KB content, ~700 ms). So this
    is **not** a browser-escalation gap — it is a stale corpus fixture.
  - The one real (minor) bug it exposed: the handler's authoritative
    `not_found` was overwritten by the downstream `length_floor`, so the
    agent saw a retry-looking failure instead of "definitively gone." Fixed
    by the `handler-verdict-precedence` change.
- **Corpus fixture weakness.** Two stale corpus URLs inflate the low
  comments-class scores: `reddit-comments` points at a deleted post, and
  `hn-item` points at `id=40000000`, a placeholder id. Swap both for
  known-live discussions before the next run.

## Contract failures

None. Both a2web systems: 10/10 conformant on the debug=False envelope and
10/10 on the debug=True envelope.

## Fetch errors (5 of 30 cells)

- `reddit-listing` × webfetch — empty model response (Reddit served
  WebFetch a page Haiku could not answer from).
- `reddit-comments` × a2web_detail / a2web_extract — `length_floor`, raw
  tier. Root cause: the corpus URL is a deleted post (see above) — a stale
  fixture, not a product gap.
- `medium-tag` × webfetch — HTTP 403 (gated).
- `spa-react-dev` × webfetch — HTTP 404 (SPA, no JS render).

## Takeaways

1. `ask` (server-side extraction) is the right default — it wins every
   axis at once and costs ~430 envelope tokens.
2. The benchmark confirms a2web's core thesis: WebFetch fails on gated and
   JS-rendered pages; a2web does not.
3. Follow-up for the backlog: refresh the two stale corpus fixtures —
   `reddit-comments` (deleted post) and `hn-item` (placeholder id) — with
   known-live discussions. The verdict-precedence bug the deleted post
   surfaced is already fixed (`handler-verdict-precedence`).

# Run 2 — `cascade-decision-log` validation

Re-run after the `cascade-decision-log` change (append-only observation log +
`resolve_verdict` projection + `decide_next` pure planner + pure-executor
orchestrator). Same corpus, now with the two stale fixtures refreshed
(`reddit-comments` → live `ddlc78` thread, `hn-item` → live `id=9224`).
Run: 30 cells, $2.33, 5m17s wall. Run dir: `eval/runs/2026-05-22_152935/`.

## Headline — four axes, per system

| System | quality | env tokens | clarity | contract | next_links |
|---|---|---|---|---|---|
| webfetch_baseline | 3.10 | 175 | 3.13 | n/a | n/a |
| a2web_detail | **4.50** | 3019 | 1.60 | 10/10 | 4.00 |
| a2web_extract | 4.30 | 468 | 3.20 | 10/10 | 5.00 |

## Did the cascade refactor regress anything?

**No.** The change explicitly promised no wire/envelope change, and the
benchmark confirms it: **contract 10/10 for both a2web systems on the
debug=False AND debug=True envelopes** — identical to Run 1. The verdict
now derives from `resolve_verdict(observation log)` instead of a mutable
slot; the response shape is unchanged.

- Answer quality is stable within live-network noise: `a2web_detail`
  4.10 → 4.50, `a2web_extract` 4.40 → 4.30. Reach 10/10 (detail) and
  9/10 (extract).
- **Comments class recovered.** Was ≤2.5 in Run 1 (the deleted-post
  fixture). With the live `ddlc78` thread + `id=9224` item, the comments
  class scores **5.00** for `a2web_extract` and **3.50** for detail. The
  Run-1 `reddit-comments` fetch-error is gone — it fetches cleanly through
  the Reddit handler. This validates both the corpus refresh and the
  `no_match` split (matched-but-failed handlers now surface real verdicts).
- Gated + SPA edge holds: WebFetch still **0.0** on `medium-tag` (403) and
  `react.dev` (404); a2web **5.0** on both.

## One new soft spot

- **`gh-trending` × a2web_extract scored 0** (judge_reached=False), while
  `a2web_detail` handled the same URL at 3.0. The fetch itself was
  contract-conformant (468 envelope tokens, 10/10 contract still holds) —
  this is an extraction-quality miss on the GitHub trending listing, not a
  cascade or envelope fault. Single-URL judge miss on a 10-URL live run;
  watch it next run before treating it as a real regression.

## Contract failures

None. Both a2web systems: 10/10 on debug=False and 10/10 on debug=True.

## Takeaway

The `cascade-decision-log` change is output-neutral: same envelope, same
contract conformance, quality stable within noise. The structural win
(verdict-clobber bug class eliminated, escalation policy unified in one
pure planner) carries no output-quality cost. Safe to archive.

# Run 3 — `generic-record-extraction` validation

Re-run after the `generic-record-extraction` change (recall-based escalation
trigger + multi-source extraction ladder + structural record extractor).
Run: 30 cells, $2.31, 5m30s. Run dir: `eval/runs/2026-05-22_165504/`.

## Headline — four axes, per system

| System | quality | env tokens | clarity | contract | next_links |
|---|---|---|---|---|---|
| webfetch_baseline | 3.00 | 176 | 2.88 | n/a | n/a |
| a2web_detail | 4.80 | 3311 | 1.80 | 10/10 | 4.00 |
| a2web_extract | **4.90** | 503 | 3.30 | 10/10 | 4.00 |

## The headline result: gh-trending 0 → 5

`gh-trending` × `a2web_extract` — the cell that triggered this whole
investigation — went from **0/0/0** (Run 2) to **5/5/5**. The structural
record extractor located the `article.Box-row` repo-card region, rendered the
bounded subtree to link-preserving markdown, and the extraction LLM produced
all 5 repos with `owner/name` slugs. `a2web_detail` on the same URL: 3 → 5.

This lifted the whole **listing class**: `a2web_extract` 3.50 → **5.00**,
`a2web_detail` 4.50 → **5.00**.

## Overall movement vs Run 2

- `a2web_extract` overall **4.30 → 4.90**, reach **9/10 → 10/10**.
- `a2web_detail` overall 4.50 → 4.80.
- Every URL class is now ≥ 4.0 for both a2web systems; gated + SPA still 5.0,
  WebFetch still 0.0 on both.

## No regression

**Contract 10/10** for both a2web systems on the debug=False AND debug=True
envelopes — the change kept its "no envelope change" promise. The recall-based
trigger replaced the absolute-length gate without disturbing any well-extracted
page: clean articles still score 4.5–5.0.

## Watch item

- `gh-trending` `next_links` is now populated (it was empty in Run 2 — generic
  listing pages produced no candidates) but scored only **2/5**. The record
  extractor feeds candidates through the `ask`-mode LLM re-rank; the modest
  score suggests the re-rank or the candidate `reason` field ("listing record")
  is weak. Worth a look before treating `next_links` on generic listings as
  done — extraction quality is solved, candidate *ranking* is the soft spot.

## Takeaway

`generic-record-extraction` closes the listing-page archetype gap. The cell
that scored 0 now scores 5; the listing class is maxed; no envelope or
contract regression. The one open thread is `next_links` ranking quality on
generic (non-handler) listings.
